## Purpose

This strategy formally separates **version creation** (technical state) from **release communication** (business event). Not all versions are communicated. Not all versions require stakeholder attention. This policy protects business communication from version inflation while maintaining technical precision.

---

## Core Principle

The Spektra engine follows a **high-frequency versioning model** where:
- Every merged change creates a version
- Versions document the current state
- Communication is selective based on impact

**Deterministic versioning does NOT require deterministic communication.**

---

## Definitions

### Version
A **technical state identifier** created automatically by semantic-release.
- Represents a point in the codebase history
- Always documented in CHANGELOG
- Always tagged in Git
- Does NOT require human communication

**Example**: `engine-v1.2.8`

### Release Announcement
A **communicated event** where stakeholders are informed of a version's existence and significance.
- Selective based on impact
- Requires human decision
- May cover multiple versions
- Targets specific audiences

**Example**: "Engine 1.3.0 introduces new theming capabilities"

## Communication Decision Matrix

| Version Type | Features | Breaking Changes | Bug Fixes | Maintenance | Communication Required? |
|--------------|----------|------------------|-----------|-------------|------------------------|
| **MAJOR** | Any | YES | Any | Any | **MANDATORY** |
| **MINOR** | YES | No | Any | Any | **RECOMMENDED** |
| **PATCH** | No | No | YES | Any | **CONDITIONAL** |
| **PATCH** | No | No | No | YES | **OPTIONAL** |

### Decision Rules

#### MANDATORY Communication
- All MAJOR versions
- Any version with breaking changes
- Any version that affects consumer APIs

**Rationale**: Consumers must be aware of changes requiring action.

#### RECOMMENDED Communication
- MINOR versions with new features
- Bug fixes that resolve known issues
- Performance improvements with measurable impact

**Rationale**: Consumers benefit from awareness of new capabilities.

#### CONDITIONAL Communication
- PATCH versions with critical bug fixes
- Security-related PATCH versions
- Bug fixes to recently reported issues

**Rationale**: Depends on severity and urgency.

#### OPTIONAL Communication
- Maintenance-only PATCH versions
- Internal refactoring versions
- Dependency updates without API impact

**Rationale**: Low external value; CHANGELOG is sufficient.

#### Stakeholder Announcements
**Audience**: Business stakeholders, product owners

## Batched Communication

### When to Batch
Multiple PATCH versions with only Maintenance entries can be batched into a single communication.

**Example**:
```
Versions 1.2.8, 1.2.9, 1.2.10 (Jan 8-10):
- Routine dependency updates
- Internal performance improvements
- No API changes

Current stable: v1.2.10
```

### When NOT to Batch
- Versions with breaking changes (always communicate immediately)
- Versions with user-facing bug fixes (communicate individually)
- MINOR versions (each deserves dedicated communication)

## Silent Releases

A **silent release** is a version that:
- Is documented in CHANGELOG
- Is tagged in Git
- Is available for consumption
- Is NOT actively communicated to stakeholders

**Silent releases are valid and expected.**

### Criteria for Silent Release
- Maintenance-only changes
- Internal refactoring
- Dependency updates without consumer impact
- CI/CD improvements

### Example
```markdown
## [1.2.8] - 2026-01-09

### Maintenance
- Updated React to 18.3.1
- Improved build performance
- Updated CI pipeline configuration
```

**Communication**: None required. Version exists for audit trail and state documentation.

---

## Communication Timing

### Immediate (within 1 hour)
- MAJOR versions with breaking changes
- Security-related patches
- Critical bug fixes

### Same Day
- MINOR versions with features
- Notable bug fixes

### Weekly Digest (optional)
- Multiple maintenance PATCH versions
- Low-priority updates

## Engine as Platform Context

The Spektra engine is an **internal platform**, not a public library. Communication strategy reflects this:

### Internal vs External
- **Internal**: Direct communication possible; synchronous updates feasible
- **External**: Would require broader announcements; not applicable yet

### Consumer Awareness
- Consumer projects depend on engine via package.json
- Projects can specify version ranges or pin versions
- Communication helps projects decide when to update

### Versioning Rhythm
- High-frequency technical versioning
- Low-frequency business communication
- This separation is intentional and healthy

## Anti-Patterns

### âŒ Announce Every Version
**Problem**: Communication fatigue; stakeholders ignore announcements.

**Correct Approach**: Selective communication based on impact.

### âŒ Skip Documenting Silent Releases
**Problem**: Audit trail gaps; unclear version history.

**Correct Approach**: Document all versions in CHANGELOG; communicate selectively.

### âŒ Delay Breaking Change Communication
**Problem**: Consumer projects break unexpectedly.

**Correct Approach**: Immediate communication for MAJOR versions.

### âŒ Over-Explain Maintenance Versions
**Problem**: Confusion about why maintenance versions exist.

**Correct Approach**: Document in CHANGELOG; do not communicate unless asked.

## Metrics and Review

### Communication Effectiveness Metrics
- Stakeholder awareness of breaking changes (target: 100%)
- Time to consumer project updates (target: < 2 weeks for MAJOR)
- False positive communications (target: < 10% of announcements)

### Quarterly Review
Platform team reviews:
- Communication patterns
- Stakeholder feedback
- Version frequency vs communication frequency
- Adjustment needs

---

## Policy Stability

This strategy is considered **stable and enforced**. Changes require:
- Platform team consensus
- Documentation update
- Stakeholder notification
